个人博客系统：功能 → 知识 → 阶段

你的真实项目。每节课的产出都长在这些功能上。 这张表回答两个问题：「这个功能要学什么」和「为什么得先补 CSS/JS/TS」。

一句话定义

一个带后端 API 的个人博客：访客能浏览/搜索文章，作者能登录发文章，管理员能管权限。 它麻雀虽小，却恰好覆盖了你列的全部知识点 —— 这正是选它做落地载体的原因。

技术栈 · 一处待决策

前端：React（与你「先 React」一致）。博客强需 SEO，倾向用 Next.js（React 的全栈框架，自带 SSR/SSG）而非纯 SPA。

后端：延后到第 4–5 阶段再定 —— Java Spring Boot（你的强项） 还是 TS 全栈（better-auth 原生契合、SEO 一体化）。现在不急着选，等你看得懂 React 再评估。 在此之前，所有 CSS / JS / TS 课程与这个决策无关。

为什么必须先补 CSS / JS / TS

把你列的每个「花哨功能」往下拆一层，你会看到它们全都站在同一个地基上。 地基不牢，功能就是空中楼阁 —— 这正是你过去「粘得出、改不动」的根因。

页面动效

底下是 CSS transition / transform / keyframes，再用 JS 控制触发时机。 不会盒模型和定位，元素都不知道往哪动。→ 第 ② 阶段。

SEO

底下是 语义化 HTML（article/nav/h1）+ 渲染模式。 搜索引擎读的是结构，不是样式。→ 贯穿 ②③，定型于 ⑤。

better-auth 认证

底下是 表单 + 受控状态 + HTTP 会话 + 受保护路由。认证库只是把这些粘起来。 不会表单和异步，认证逻辑就是黑盒。→ ③ JS + ⑤ React。

RBAC 动态权限

底下是 条件渲染 + 全局状态管理 + API 鉴权。「这个按钮只给管理员看」 本质是 if (user.role === 'admin') 的渲染分支。→ ⑤ React。

全文搜索

底下是 受控输入 + 防抖 + 异步(Promise/async) + 列表渲染。 搜索框每敲一字就发请求会打死后端，防抖是 JS 异步的经典应用。→ ③ JS + ⑤ React。

心法

框架和这些库（better-auth、framer-motion…）都是把基础能力封装好的产物。 你要学的不是「怎么调某个库」，而是「这个库底下在替我做什么」—— 弄懂地基，任何库都是一层薄壳。 这也是为什么你的 Java 经验是优势：你在后端早已看透「框架是封装」，前端同理。
